#rest #http #contracts

Design de APIs RESTful

TL;DR: Consistência é rei. Se usa camelCase num endpoint e snake_case noutro, a Developer Experience (DX) será terrível.
Padrões de Nomenclatura (Paths, Queries, Headers)
  • Paths: Sempre plural e kebab-case. Ex: /v1/negociacoes-cdb.
  • Query Params: Sempre camelCase. Ex: ?dataInicio=2024-01-01.
  • Headers Customizados: Padrão Hyphenated-Pascal-Case. Ex: X-Correlation-ID.
  • Hierarquia Máxima: 3 níveis de profundidade. Ex Aceitável: /clientes/{id}/enderecos/{id}/telefones.
Tipagem, Datas e Nulos
  • Datas: Siga a RFC 3339. Sufixo de timezones é obrigatório. Ex: 2024-05-20T14:30:00Z. Prefixo em/at para o campo.
  • Booleanos: Sempre usar verbos interrogativos is/has/foi/pode. Ex: `isAtivo`, `foiAprovado`.
  • Coleções (Listas): NUNCA retorne null. Se não houver dados, retorne um array vazio [].
  • Enums: Sempre serialize como `string` constante no JSON de retorno. Evita quebras se a ordem mudar no C#.
Status Codes de Resiliência
  • 202 Accepted: O padrão para processamento assíncrono (Webhooks, mensagens enviadas para filas).
  • 415 Unsupported Media Type: O cliente tentou enviar um Content-Type que a API não aceita.
  • 422 Unprocessable Entity: JSON sintaticamente correto, mas falhou nas regras de negócio (Ex: saldo insuficiente).
#architecture #solid #patterns

Arquiteturas e Princípios S.O.L.I.D.

Hexagonal, Clean, Vertical Slice & SOA

Hexagonal (Ports & Adapters): Domínio no centro. Comunicação com o mundo externo ocorre via Portas (Interfaces no Domínio) e Adapters (Implementações na Infra/API). Divide-se entre Primary/Driving (API) e Secondary/Driven (Base de Dados).

Clean Architecture: Baseia-se na "Regra de Dependência" (setas apontam para dentro). Domain -> Application -> Infra. Gera muito boilerplate (DTO mappings), mas é excelente para domínios massivos.

Vertical Slice Architecture: Em vez de camadas horizontais (Pasta Repositories, Controllers), fatia-se por Feature. Um ficheiro contém Endpoint, Command, Handler e Query. Alto acoplamento local, mas baixo acoplamento global. (Uso intensivo de MediatR).

SOA vs Microservices: SOA usa serviços "gordos" conectados por um ESB pesado, frequentemente partilhando base de dados. Microservices são focados num único contexto (Bounded Context) e cada um possui a sua própria base de dados independente.

CQRS, BFF e Circuit Breaker (Padrões de Alta Escala)

CQRS (Command Query Responsibility Segregation): Separa as operações de Escrita (Commands - complexas, validam regras, usam EF Core) das de Leitura (Queries - rápidas, retornam DTOs, usam Dapper). Permite escalar bases de dados de forma independente.

BFF (Backend for Frontend): Um API Gateway dedicado a uma interface específica (ex: um BFF para Mobile, outro para Web). Aglutina chamadas de vários microsserviços e entrega o JSON limpo para o front-end, reduzindo as idas e vindas de rede (chattiness).

Circuit Breaker: (via Polly). Se uma API externa parceira falha X vezes seguidas, o circuito "abre". O seu sistema pára de tentar chamar a API externa por uns segundos, falhando rapidamente para não esgotar as *Thread Pools* do seu próprio servidor.

S.O.L.I.D. Direto ao Ponto
S (Responsabilidade Única): Uma classe deve ter apenas 1 motivo para mudar.
O (Aberto/Fechado): Fechado para alteração, aberto para extensão (Evite `ifs`, use polimorfismo/interfaces).
L (Substituição de Liskov): A classe filha nunca pode quebrar o contrato ou lançar exception inesperada se substituir a classe pai.
I (Segregação de Interfaces): Quebre interfaces "Deus". Ninguém deve ser forçado a implementar o que não usa.
D (Inversão de Dependência): Módulos de alto nível não dependem de baixo nível. Ambos dependem de abstrações (Interfaces). A base da Injeção de Dependência no .NET.
Complexidade Ciclomática e Big O Notation

Complexidade Ciclomática: Mede os caminhos independentes do código. Quantos `if/else`, `switch`, `for` tem num método? Acima de 10, o código torna-se impossível de testar na totalidade. Solução: Extract Method ou usar o padrão Strategy.

Big O Notation (Performance): Avalia o tempo de execução mediante o crescimento dos dados (N).
- O(1): Acesso direto a um Dictionary/HashSet (Perfeito).
- O(N): Iterar um `IEnumerable` ou Lista com `foreach` (Normal).
- O(N²): Um `foreach` dentro de outro `foreach` iterando as mesmas coleções (Catastrófico).

#ddd #domain

Domain-Driven Design (DDD)

Domínio Anémico vs Rico

Anémico: Entidades são sacos de dados (getters/setters abertos). Regras, `ifs` e cálculos moram no `Service`. Fere o encapsulamento.

Rico: A entidade protege o seu estado. Setters são privados. A alteração de estado dá-se via métodos de negócio. Em vez de fatura.Status = Pago, usa-se fatura.Pagar().

Value Objects & Aggregate Roots

Aggregate Root: Grupo de entidades lideradas por uma Raiz. Acesso aos filhos só passa pela Raiz. (O Pedido gere os itens e garante que a soma está correta).

Value Object: Representa um aspeto tipado, sem identidade. Imutáveis. Igualdade é por valor. Constroem-se e validam-se no construtor. No C#, use record.

ACL (Anti-Corruption Layer) vs Adapter

Adapter: Converte interfaces técnicas (ex: envolver o Serilog num `ILogger` interno).

ACL: Camada entre o sistema novo e um sistema Legado. Traduz o "lixo/jargão" do legado para o modelo limpo do seu Bounded Context logo na entrada.

#csharp #dotnet #memory

C# & .NET Deep Dive

Atenção Staff: A maioria das falhas de performance no .NET advém do mau uso do Entity Framework e de alocações excessivas na memória devido a coleções e strings incorretas.
Coleções: IEnumerable vs IQueryable vs List vs Array

IEnumerable<T>: O nível mais abstrato. Iteração forward-only (avançar apenas). O código é processado na memória da aplicação via Lazy Evaluation. É o tipo ideal para retornos em Interfaces.

IQueryable<T>: O motor do EF Core. Traduz o LINQ para Árvores de Expressão (SQL) e roda o filtro dentro da base de dados. Se aplicar um `.ToList()` prematuramente, o `IQueryable` converte-se num `IEnumerable`, trazendo milhares de linhas para a memória da API e causando Out of Memory.

List<T>: Um array dinâmico alocado na Heap. Muito útil para manusear dados (Add/Remove) na aplicação. Cuidado: Quando a lista enche o seu buffer interno, ela aloca um novo array com o dobro do tamanho e copia os dados originais (custo de performance).

Array (T[]): Tamanho fixo. É a coleção com melhor performance do .NET devido à alocação contígua na memória e ausência de overhead.

Dictionary e HashSet: Essenciais para a otimização algorítmica. O `HashSet` valida se algo já existe na lista em O(1), enquanto o `.Any()` numa `List` ou `IEnumerable` fá-lo em O(N).

Ecossistema de Configurações (.NET DI & Options)

O motor de injeção e definições do .NET Core.

IServiceCollection: O contentor de Injeção de Dependência (DI) onde regista os seus serviços (`AddScoped`, `AddSingleton`, `AddTransient`).

appsettings.json vs Environment: O .NET carrega o `appsettings.json` base. De seguida, sobrescreve chaves se existir um `appsettings.{Environment}.json` (ex: Production). Variáveis de Ambiente e Command Lines sobrescrevem tudo no topo da hierarquia.

IConfiguration: Usado na class `Program`/`Startup` para ler *strings* avulsas. Anti-pattern: Injetar `IConfiguration` em classes de negócio; usa-se o Options Pattern.

O Options Pattern:
  • IOptions<T>: É Singleton. Lê as chaves no arranque e guarda na cache.
  • IOptionsSnapshot<T>: É Scoped. Relê as configurações a cada Request HTTP (útil se o `appsettings` for alterado sem reiniciar a API).
  • IOptionsMonitor<T>: Singleton que atualiza os valores em tempo de execução via eventos (ideal para *Feature Flags* no RabbitMQ/Redis).
Tipos no C#: Class vs Struct vs Record

Class (Reference Type): Alocada na Heap. Passada por referência. É o padrão para Entidades de Domínio (Aggregate Roots) onde a identidade (ID) importa mais que o valor.

Struct (Value Type): Alocada na Stack (zero pressão no GC). Passada por cópia. Ideal para pequenas estruturas matemáticas (ex: `Vector3`, `DateTime`).

Record (Reference Type com Value Equality): Alocada na Heap, mas o compilador injeta métodos de igualdade baseados no valor das propriedades. Imutáveis por defeito (`init`). Perfeitos para DTOs, Commands, Events e Value Objects.

Readonly Record Struct: O ápice da performance. Combina a alocação na Stack (Struct) com a imutabilidade e igualdade do Record. Usado em código que processa milhões de operações por segundo.

Span<T>, Memory<T> e StringBuilder (Zero-Allocation)

StringBuilder: Usa um buffer pré-alocado. Modifica a string no local (In-place). Use sempre dentro de loops para evitar criar milhares de cópias de Strings na Heap.

ReadOnlySpan<char>: Um Ref Struct alocado na Stack. Atua como uma "janela" apontando para a string original. Ao extrair partes com `Slice`, não cria NENHUM objeto na Heap.

#sql #nosql #dapper

Bases de Dados & Persistência

EF Core vs Dapper vs ADO.NET

EF Core: Abstração máxima. Controlo de transações (Unit of Work). Ideal para Escrita (Commands) onde o Domínio dita as regras e atualizações controladas por Trackers.

Dapper: Micro-ORM. Baixa alocação de memória. Ideal para Leitura (Queries), relatórios e arquiteturas CQRS.

ADO.NET: Drivers puros. Apenas para Bulk Inserts massivos onde os milissegundos importam.

Transações, ACID e Locks

ACID: Atomicidade (tudo ou nada), Consistência (regras não violadas), Isolamento, Durabilidade.

Lock Pessimista: `SELECT FOR UPDATE`. Trava a linha fisicamente. Risco altíssimo de Deadlock. Solução: transações extremamente curtas.

Lock Otimista: Não trava a BD. Usa coluna `RowVersion` (Timestamp). Se duas threads gravam ao mesmo tempo, a segunda sofre `ConcurrencyException`. Padrão Web (não bloqueante).

Índices e NoSQL (Teorema CAP)
  • Table Scan vs Index Scan/Seek: Um Table Scan lê a tabela inteira por falta de índices, matando a BD em produção. O Index Seek usa B-Trees para ir direto à linha no disco.
  • Teorema CAP (NoSQL): Consistência (C), Disponibilidade (A), Tolerância a Partição (P). Escolha entre CP (Mongo) ou AP (DynamoDB).
  • Consistência Eventual (Eventual Consistency): Num NoSQL distribuído ou arquitetura baseada em eventos (Kafka), a gravação num nó demora milissegundos a propagar. A arquitetura deve assumir que os dados podem estar temporariamente desatualizados no instante exato após a gravação.
#tests #xunit #quality

Pirâmide de Testes

A Pirâmide: Unit, Integration & E2E

Unitários (70%): Testam Domínio e Regras matemáticas isoladas. Usam Mocks para isolar I/O. Milhares rodam em segundos no GitHub Actions.

Integração (20%): O novo normal do .NET. Esqueça "InMemoryDb". Use Testcontainers para subir um Docker real do Postgres durante o teste do Controller, testando as queries e restrições reais da base de dados, destruindo o container no fim.

E2E - End to End (10%): Ferramentas como Playwright que abrem um Chromium sem interface e clicam na aplicação real. Lentos e suscetíveis a falhas visuais. Usados apenas em Core Journeys.

A Tríade AAA & Test Doubles (Mocks/Stubs)

Padrão Ouro: Arrange (Preparar dados), Act (Chamar método), Assert (Validar asserções com FluentAssertions).

Stub: Objeto fixo. Prove um "OK" falso para o teste do método A avançar livremente.

Mock: Focado em Comportamento. O teste Assert passa se, e só se, o objeto fingido foi chamado exatemente as vezes estipuladas. (Ex: "Valide se o `PublishKafkaAsync()` foi chamado 1 vez").

#aws #k8s #docker

Cloud, Containers & Networking

Kubernetes (K8s) & LocalStack

K8s: Orquestrador Declarativo. Pod: Unidade efémera com containers. Deployment: Garante as réplicas e o estado desejado. Service: O Load Balancer interno de IP estático que mapeia para Pods.

LocalStack: Framework em Docker para simular serviços geridos da AWS (SQS, SNS, S3, Dynamo) offline na máquina do Dev.

AWS VPC & Security (O Segredo das Entrevistas)

VPC & Subnets: A sua rede privada. A Subnet Pública (com Internet Gateway) hospeda o Load Balancer. A Subnet Privada (sem internet) abriga os workers e base de dados, por segurança.

NACL (Subnet): Stateless. Se abre uma porta de Entrada, tem explicitamente de abrir portas para a Saída. Avaliado na fronteira da Subnet.

Security Group (Instância): Stateful. Avaliado ao nível do Recurso (ex: a EC2). Se permitiu tráfego de entrada via HTTP(80), ele lembra a sessão e garante a resposta de volta automaticamente.

Docker Basics Cheat Sheet

docker build -t app . -> Gera imagem a partir do Dockerfile.

docker run -p 8080:80 app -> Roda e mapeia portas.

docker exec -it [ID] /bin/bash -> Acesso SSH interativo ao container.

Dica Sênior: Substitua as imagens base pesadas de SO por imagens Alpine (5MB) ou Distroless da Google. Menor superfície de ataque e menos CVEs na pipeline de segurança.

#git #github #workflow

Git Strategies & Configs

Git Global Configs (The Senior Setup)

Automatizar o Git economiza tempo e padroniza a equipa sem esforço.

Template de Commit Global: Cria um `.gitmessage` pré-preenchido para forçar o formato Conventional Commits no terminal ou VS Code.
git config --global commit.template ~/.gitmessage

Auto Rebase: Evita o "Merge branch 'main' of..." sujo ao efetuar pulls, refazendo o histórico numa linha reta.
git config --global pull.rebase true

Editor Padrão (Vim):
git config --global core.editor "vim"

Branching Models & Conventional Commits

Git Flow: Burocrático (`develop`, `release`, etc). Considerado um anti-padrão para startups modernas e SaaS.

GitHub Flow: Simples: Master protege a Prod -> Cria Feature Branch -> Pull Request -> CI/CD e Deploy.

Trunk-Based Development: Nível Staff. Desenvolvedores efetuam *push* para a Master várias vezes ao dia. Requer Feature Flags obrigatórias para esconder código incompleto na produção.

Conventional Commits: feat: (adiciona feature), fix: (corrige bug), chore: (manutenção). Crucial para gerar changelogs e determinar automaticamente a versão do software (*Semantic Versioning*: Major.Minor.Patch).

#linux #vim #cli

Linux & Vim Survival Guide

TL;DR: O terminal é a IDE em produção. Dominar ferramentas POSIX e o Vim separa juniores que entram em pânico de seniores que resolvem quedas via SSH.
Vim: O Guia de Sobrevivência

Modos: Esc (Modo Normal - Comandos), i (Modo Insert - Texto).

Gravar e Sair: :w (grava), :q! (sai forçado sem gravar), :wq ou ZZ (grava e sai).

Navegação Ninja: h j k l (Esquerda, Baixo, Cima, Direita). w (pula para a próxima palavra). 0 (início de linha), $ (fim de linha).

A Dica Sênior: ciw (Change In Word) - Numa palavra longa, posicione o cursor sobre ela, digite "ciw" para apagar a palavra e entrar no modo Insert instantaneamente.

A Trindade POSIX: grep, awk, sed

grep: grep -rnw '/var/log' -e 'Exception' busca de forma recursiva a palavra exata exibindo os números da linha.

awk: Excelente processador de colunas de texto. Exemplo: ps aux | awk '{print $2}' extrai apenas o número dos PIDs da segunda coluna.

sed: sed -i 's/antigo/novo/g' config.json para substituir variáveis de forma silenciosa e in-place em ambientes CI/CD.

Troubleshooting de Rede & Permissões

Portas Abertas: Use ss -tuln ou o clássico netstat -tulpn para perceber o que roda internamente no servidor.

Matar Porta Trancada: kill -9 $(lsof -t -i:8080) liquida sumariamente o processo que bloqueia o seu recomeço de API na 8080.

O Terror das Permissões: Nunca aplique chmod 777. Use chmod 644 em ficheiros (Dono L/E, Resto só Lê) e chmod 755 em diretórios/scripts bash.

The 36 Q&A Vault

Todas as 36 perguntas de entrevistas Sênior/Staff respondidas de forma cirúrgica.

1. C# & .NET Core Internals

Q1, 34, 35: Task vs Thread? ValueTask vs Task?

Thread: Criação pesada (~1MB). `Task` é promessa que executa na ThreadPool via StateMachine gerada pelo `async/await`.

Task vs ValueTask: `Task` aloca na Heap (GC). `ValueTask` é struct (na Stack). Use ValueTask se o método quase sempre retorna de forma síncrona/cacheada.

Q4, 5, 33b: Heap vs Stack? GC? JIT?

Tipos: Value Types (struct) vão pra Stack. Reference Types (class) vão pra Heap.

GC: Limpa a Heap. Dividido em Gen(0, 1, 2). Coletas na Gen2 causam "Stop the World".

JIT: Compila o código intermediário (IL) para binário de máquina em Runtime.

Q3: Injeção de Dependência e Ciclos

Transient: Nova sempre. Scoped: Mesma instância no request (Ideal para EF Core). Singleton: Instância global.

2. Banco de Dados e Concorrência

Q9, 26, 30: O que é ACID? Lock Otimista vs Pessimista?

ACID: Atomicidade, Consistência, Isolamento, Durabilidade.

Otimista: Usa `RowVersion`. Não trava DB. Pessimista: Trava a linha no banco, gera Deadlock.

Q25, 27: Truncate vs Delete? Resolver Deadlock?

Truncate: Apaga tudo, instantâneo, não aceita Rollback. Delete: Grava no Transaction Log, aceita Rollback.

Deadlock: Solução: Acesse recursos na mesma ordem, mantenha transações curtas.

Q20, 28, 29, 33: Índices? SQL vs NoSQL? Pools?

Índices: Clusterizado (Ordena disco, 1 por tabela). Não-Clusterizado (Cria um sumário).

Connection Pool: Reaproveita conexões TCP/TLS com o banco.

3. Arquitetura, Segurança e Resiliência

Q6, 10, 24: Resiliência Externa e Filas (Idempotência)

Circuit Breaker: Polly. Falhou 3x? Abre o circuito.

Idempotência: Garante que processar a mensagem do Kafka 2x não duplique a compra. Erros vão pra DLQ.

Q21, 22: Cancelamento e Saga Pattern

CancellationToken: Cancela Request = Rollback no EF Core.

Saga Pattern: Transações de Compensação para reverter chamadas externas se houver falha.

Q7, 8, 18, 19: Proteger API e Observabilidade

AuthN vs AuthZ: Autenticação (Quem é). Autorização (O que pode fazer).

Proteção: Rate Limiting por IP para barrar DDOS. OpenTelemetry para tracing distribuído.

Q11, 12, 14: Cloud e Padrões (Patterns)

Terraform: IaC (Infrastructure as Code).

Patterns Chaves: Strategy, Decorator, Outbox Pattern (Salvar BD e publicar Evento de forma atômica).

4. Soft Skills & Clean Code

Q13, 31, 32, 36: Testes e Coesão/Acoplamento

Coesão e Acoplamento: Alta Coesão (Classe focada) e Baixo Acoplamento (Interfaces).

Código Bom: Legível, expressa o negócio, testável. "Apenas compilar" não basta.

Q15, 16, 17, 23: Dívida Técnica e Squads

Dívida Técnica: Visibilidade no Jira. Usar a Regra do Escoteiro ou alocar 20% da Sprint para refatoração core.
