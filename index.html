#rest #http #contracts

Design de APIs RESTful

TL;DR: Consistência é rei. Se você usa camelCase num endpoint e snake_case em outro, o Developer Experience (DX) dos seus consumidores será terrível.
Padrões de Nomenclatura (Paths, Queries, Headers)
  • Paths: Sempre plural e kebab-case. Ex: /v1/negociacoes-cdb.
  • Query Params: Sempre camelCase. Ex: ?dataInicio=2024-01-01.
  • Headers Customizados: Padrão Hyphenated-Pascal-Case, geralmente prefixado. Ex: X-Correlation-ID.
  • Hierarquia Máxima: 3 níveis de profundidade. Passou disso, repense. Ex Aceitável: /clientes/{id}/enderecos/{id}/telefones.
Tipagem, Datas e Nulos
  • Datas: Siga a RFC 3339 rigorosamente. Sufixo de timezones é obrigatório. Ex: 2024-05-20T14:30:00Z. Prefixo em/at para o campo.
  • Booleanos: Sempre usar verbos interrogativos is/has/foi/pode. Ex: `isAtivo`, `foiAprovado`.
  • Coleções (Listas): NUNCA retorne null. Se não houver dados, retorne um array vazio [].
  • Enums: Sempre serialize como `string` constante no JSON de retorno. Se a ordem numérica do Enum mudar no backend, não quebra os clientes.
Status Codes de Resiliência
  • 202 Accepted: O padrão para processamento assíncrono. Retorne isso quando a requisição for para uma fila (Worker) ou Webhook e o resultado virá depois.
  • 415 Unsupported Media Type: O cliente tentou enviar um Content-Type que a API não aceita (ex: enviou XML, mas você só aceita `application/json`).
  • 422 Unprocessable Entity: O JSON está perfeito sintaticamente, mas falhou nas regras de negócio/validações de domínio (ex: Tentou transferir dinheiro, mas o saldo é menor que o valor).
#architecture #solid #patterns

Arquiteturas e Princípios S.O.L.I.D.

Hexagonal, Clean, Vertical Slice & SOA

Hexagonal (Ports & Adapters): Domínio no centro. Comunicação com o mundo externo ocorre via Portas (Interfaces no Domínio) e Adapters (Implementações na Infra/API). Divide entre Primary/Driving (quem chama a API) e Secondary/Driven (o DB que a API chama).

Clean Architecture: Baseia-se na "Regra de Dependência" (setas apontam pra dentro). Domain -> Application -> Infra. Gera muito boilerplate (DTO mappings), mas excelente para domínios massivos.

Vertical Slice Architecture: Em vez de camadas horizontais (Pasta Repositories, Pasta Controllers), você fatias por Feature. Um arquivo contém Endpoint, Command, Handler e Query. Alto acoplamento local, mas baixo acoplamento global. (Usa muito MediatR).

SOA vs Microservices: SOA usa serviços "gordos" conectados por um ESB pesado, geralmente dividindo banco de dados. Microservices são focados em um único contexto (Bounded Context) e cada um possui seu próprio banco de dados independente.

S.O.L.I.D. Direto ao Ponto
S (Responsabilidade Única): Uma classe deve ter apenas 1 motivo para mudar.
O (Aberto/Fechado): Fechado para alteração, aberto para extensão (Evite encher de `ifs`, use polimorfismo/interfaces).
L (Substituição de Liskov): A classe filha nunca pode quebrar o contrato ou lançar exception inesperada se substituir a classe pai.
I (Segregação de Interfaces): Não crie interfaces "Deus". Quebre em interfaces menores. Ninguém deve ser forçado a implementar o que não usa.
D (Inversão de Dependência): Módulos de alto nível não dependem de baixo nível. Ambos dependem de abstrações (Interfaces). A base da Injeção de Dependência no .NET.
Engineering Principles (YAGNI, DRY, Coesão/Acoplamento)

Baixa Coesão / Alto Acoplamento: O pior cenário. Alto acoplamento: Alterar Classe A quebra Classe B. Baixa Coesão: A classe faz muita coisa sem sentido (ex: `Helper.cs` envia email, calcula imposto e faz parser de string).

YAGNI (You Aren't Gonna Need It): Não adicione complexidade prevendo o futuro. Se precisa de um Gateway, faça pra um. Não crie uma Abstract Factory complexa "para caso entre o PayPal no ano que vem".

DRY (Don't Repeat Yourself): Evite duplicar lógica de negócio core. Mas atenção: é aceitável ter o mesmo código DTO em dois microserviços diferentes para evitar Acoplamento Temporal.

#ddd #domain

Domain-Driven Design (DDD)

Domínio Anêmico vs Rico

Anêmico: Entidades são apenas sacos de dados (getters e setters abertos). Toda regra, `ifs` e cálculos moram no `Service`. Fere o encapsulamento.

Rico: A entidade protege seu estado. Setters são privados. A alteração de estado se dá via métodos que expressam comportamentos de negócio. Ao invés de fatura.Status = Pago, usa-se fatura.Pagar().

ACL (Anti-Corruption Layer) vs Adapter

Adapter: Converte interfaces técnicas (ex: envelopar o Serilog em um `ILogger` interno).

ACL: Camada entre seu sistema e um sistema Legado. Evita que o jargão e a bagunça estrutural do legado "corrompam" o seu Bounded Context moderno. Você traduz o lixo do legado para seu modelo limpo logo na entrada.

Value Objects & Aggregate Roots

Aggregate Root: Grupo de entidades lideradas por uma Raiz. Acesso aos filhos só passa pela Raiz. Ex: Você não insere um `ItemPedido` no banco. Você chama `Pedido.AdicionarItem()`, pois o Pedido deve gerenciar a invariante (ex: total da compra).

Value Object: Representa um aspecto tipado, sem identidade (ID). São Imutáveis. Igualdade é por valor (1 minuto == 60 segundos). Constroem e validam a si mesmos no construtor. No C#, use record.

public record Dinheiro
{
    public decimal Valor { get; init; }
    public string Moeda { get; init; }
    
    // Auto-validação: Nasce sempre válido
    public Dinheiro(decimal valor, string moeda) { ... }
}
Linguagem Ubíqua, Bounded Contexts & Domain Events
  • Linguagem Ubíqua: Devs e Negócios falam a mesma língua. Se chamam de "Assinante", o código não deve ter classe "User".
  • Bounded Context: O contexto delimita o significado da entidade. "Cliente" no contexto de Vendas (precisa de histórico) é diferente de "Cliente" no de Entregas (precisa só de endereço e GPS).
  • Domain Events: Acontecimentos passados que importam pro negócio. PedidoCriadoEvent é disparado para desacoplar a notificação (e-mail) da regra de criação.
#csharp #dotnet #performance

C# & .NET Deep Dive

Span<T>, Memory<T> e StringBuilder (Zero-Allocation)

O Problema: Strings em C# são imutáveis. Concatenar com `+=` em loops ou usar `.Substring()` cria incontáveis cópias inúteis na memória Heap (acionando o GC e travando CPU).

StringBuilder: Usa um buffer pré-alocado. Modifica a string no lugar (In-place). Essencial dentro de loops.

ReadOnlySpan<char>: É um Ref Struct alocado na Stack. Ele atua como uma "janela" apontando para a string original. Ao extrair partes (`Slice`), não cria NENHUM objeto na Heap. É performance pura para parsers.

Integrações HTTP: Refit vs HttpClient

Refit: Transforma APIs REST em Interfaces C# vivas. Remove todo o boilerplate de serialização e roteamento. É a abordagem primária moderna.

HttpClient + AuthHandler: Use a via nativa quando precisar interceptar o ciclo de vida (DelegatingHandler) para injetar JWTs dinamicamente, ou quando a API externa não é RESTful.

// Usando Refit limpo
public interface IViaCepApi {
    [Get("/ws/{cep}/json")]
    Task<CepResponse> ConsultarAsync(string cep);
}
#sql #nosql #dapper

Bancos de Dados & Persistência

EF Core vs Dapper vs ADO.NET

EF Core: Abstração máxima. Controle de transações (Unit of Work). Ideal para Escrita (Commands) onde o Domínio dita as regras e updates são controlados por Trackers.

Dapper: Micro-ORM. Baixa alocação de memória, velocidade bruta mapeando T-SQL para Objetos. Ideal para Leitura (Queries), relatórios pesados e arquiteturas CQRS.

ADO.NET: Drivers puros. Uso raríssimo. Apenas para Bulk Inserts massivos onde milissegundos importam.

Transações, ACID e Locks

ACID: Atomicidade (tudo ou nada), Consistência (regras/FKs não violadas), Isolamento (transações separadas), Durabilidade (gravou no disco).

Lock Pessimista: `SELECT FOR UPDATE`. Trava a linha fisicamente. Risco altíssimo de Deadlock. Solução pra deadlock: ordem de acesso unificada e transações curtas.

Lock Otimista: Não trava o BD. Usa coluna `RowVersion`. Se duas threads tentam salvar ao mesmo tempo, a 2ª sofre `ConcurrencyException`. A aplicação tenta novamente. Padrão Web.

Índices e NoSQL (Teorema CAP)
  • Table Scan vs Index Seek: Scan lê o BD inteiro pra achar um WHERE (Custo O(N) catastrófico). Seek usa a árvore B+ do índice para ir direto ao disco.
  • Teorema CAP (NoSQL): Consistência (C), Disponibilidade (A), Partição (P). Na nuvem, P é lei. Você escolhe entre CP (Mongo) ou AP (Dynamo/Cassandra).
  • Eventual Consistency: Ao gravar num NoSQL/Tópico Kafka, as réplicas de leitura demoram ms para atualizar. Leituras imediatas pós-escrita podem retornar dados velhos. Arquitetura deve prever isso.
#tests #xunit #quality

Pirâmide de Testes no .NET

Padrão de escrita: AAA (Arrange, Act, Assert).
Nomenclatura: MetodoEmTeste_EstadoEsperado_ResultadoEsperado.
Unit, Integration & E2E

Unitários: (70%). Testam Domínio/Serviços isolados. Mocam banco de dados. Milhares rodam em segundos na CI.

Integração: (20%). Abandonamos "InMemoryDb" que não testa triggers. Usamos a lib Testcontainers que sobe um Docker real de PostgreSQL na hora do teste, roda e mata o container ao final.

End-to-End (E2E): (10%). Playwright ou Cypress. Um robô clicando na tela real. Lentos e frágeis, use apenas nas jornadas críticas.

Mocks vs Stubs vs Spies

Stub: Objeto "burro" que provê um retorno fixo para que o teste prossiga (ex: força a API fake a retornar HTTP 200).

Mock: Focado em Comportamento. O teste passa SE o Mock comprovar que o método foi chamado. (Ex: "Verifique se o `EmailService.Enviar()` foi chamado 1 vez com esse ID").

#devops #aws #k8s

Cloud, Containers & Versioning

Kubernetes (K8s) & LocalStack

K8s: Orquestrador Declarativo. Pod: Unidade efêmera que roda seu container. Deployment: Garante as réplicas. Service: O Load Balancer interno que mapeia para os Pods.

LocalStack: Permite simular SQS, SNS e S3 offline usando Docker na máquina do dev, parando de poluir a conta AWS dev.

AWS: VPC, Subnets, SG vs NACL

VPC: Rede privada virtual. Isola seus recursos.

NACL (Subnet): Stateless. Regra de entrada e saída explícitas. (Portaria do prédio).

Security Group (EC2/RDS): Stateful. Permitiu entrada na 80? A saída pela 80 volta automaticamente. (Porta do seu apartamento).

Git Strategies & Conventional Commits

Git Flow: Obsoleto para SaaS. Burocrático (`develop`, `release`, etc).

GitHub Flow: Branch da `main` -> PR -> Merge -> Deploy. Ágil.

Trunk-Based: Sêniors commitam direto na `main` várias vezes ao dia. Necessário uso de Feature Flags para código incompleto.

Conv. Commits: Padroniza `feat:`, `fix:`, `chore:`. Automatiza o Semantic Versioning (Major.Minor.Patch).

Docker Basics Cheat Sheet

docker build -t app . -> Gera imagem.

docker run -p 8080:80 app -> Sobe container expondo porta.

docker exec -it [ID] /bin/bash -> Entra no container em execução.

Dica Sênior: Use imagens base Alpine ou Distroless para derrubar o peso do container de 200MB para ~50MB e reduzir superfície de ataque.

#linux #cli #troubleshooting

Linux & CLI Mastery

TL;DR: O terminal é a sua IDE em produção. Dominar ferramentas POSIX separa os juniores que entram em pânico dos seniores que resolvem outages via SSH.
Processamento de Texto & Arquivos (A Trindade)

grep: Busca textos. grep -rnw '/var/log' -e 'Exception' busca recursivamente (r), mostrando a linha (n) e palavra exata (w).

awk: Extrai colunas. ps aux | awk '{print $2}' pega apenas os PIDs (2ª coluna).

sed: Substituição de stream. sed -i 's/antigo/novo/g' config.json altera o arquivo in-place.

find + exec: Acha e executa. find . -type f -name "*.log" -mtime +7 -exec rm {} \; deleta logs com mais de 7 dias.

Redes & Troubleshooting (Network)

Portas abertas: ss -tuln ou netstat -tulpn para listar quem está escutando em qual porta.

Testar Conexão TCP: nc -zv host.com 443 (Netcat) testa se a porta está aberta no firewall sem precisar de telnet.

Matar processo por Porta: kill -9 $(lsof -t -i:8080) mata brutalmente quem estiver travando a porta 8080.

DNS: dig +short dominio.com para debugar problemas de resolução de nome.

Serviços, Logs e Sistema

Systemd (Serviços): systemctl status/start/stop/restart meu-app.

Journalctl (Logs): Ver logs do serviço: journalctl -u meu-app -f (-f para "tail"/seguir ao vivo).

Análise de Disco/CPU: df -h (espaço em disco), du -sh * (tamanho das pastas). htop ou top para monitorar CPU e RAM dos processos.

Permissões (O básico para não usar 777)

Anti-pattern: Dar chmod 777. Isso dá permissão de Leitura(4) + Escrita(2) + Execução(1) para Dono, Grupo e Público.

Boas práticas: chmod 644 para arquivos (dono lê/escreve, outros leem). chmod 755 para diretórios/scripts (dono faz tudo, outros leem/executam).

Trocar Dono: chown usuario:grupo arquivo.txt.

The 36 Q&A Vault

Todas as 36 perguntas de entrevistas Sênior/Staff respondidas de forma cirúrgica.

1. C# & .NET Core Internals

Q1, 34, 35: Task vs Thread? ValueTask vs Task?

Thread: Criação pesada (~1MB). `Task` é uma promessa de operação assíncrona que executa na ThreadPool via StateMachine gerada pelo compilador no `async/await`, liberando a thread para outros.

Task vs ValueTask: `Task` aloca na Heap (gera trabalho pro GC). `ValueTask` é struct (na Stack). Use ValueTask se o método quase sempre retorna de forma síncrona/cacheada.

Q4, 5, 33b: Heap vs Stack? GC? JIT?

Tipos: Value Types (struct/int) vão pra Stack (Rápido, efêmero). Reference Types (class/string) vão pra Heap (precisa do GC para limpar).

GC (Garbage Collector): Limpa a Heap. Dividido em Gerações (0, 1, 2). Coletas na Gen2 causam "Stop the World".

JIT (Just In Time): Compila o código intermediário (IL) para binário de máquina no momento da execução.

Q3: Injeção de Dependência e Ciclos

Transient: Nova sempre. Scoped: Mesma instância no request (Ideal para EF Core `DbContext`). Singleton: Instância global (Cuidado com Memory Leaks e Thread-Safety).

2. Banco de Dados e Concorrência

Q9, 26, 30: O que é ACID? Lock Otimista vs Pessimista? Transação?

ACID: Atomicidade, Consistência, Isolamento, Durabilidade. Garantem segurança em transações relacionais.

Otimista: Usa `RowVersion`. Não trava. Se houver concorrência simultânea, um lança erro e a aplicação tenta de novo. Padrão Web.

Pessimista: `SELECT FOR UPDATE`. Trava a linha no banco para outros leitores/escritores. Risco de Deadlock.

Q25, 27: Truncate vs Delete? Resolver Deadlock?

Truncate (DDL): Apaga tudo, instantâneo, não aceita Rollback (geralmente).

Delete (DML): Checa constraints, grava no Transaction Log (lento), aceita Rollback e Cláusula WHERE.

Deadlock: Duas transações esperam o lock cruzado. Solução: Acesse recursos sempre na mesma ordem, mantenha transações absurdamente curtas.

Q20, 28, 29, 33: Índices? SQL vs NoSQL? Pools?

Índices: Clusterizado (Ordena fisicamente os dados, só 1 por tabela - PK). Não-Clusterizado (Cria um sumário apontando para o disco).

SQL vs NoSQL: Pergunte: Os dados são fortemente relacionados? Necessitam garantias financeiras ACID rígidas? (SQL). Ou o volume é insano e o acesso é quase sempre por Chave? (NoSQL).

Connection Pool: Pool mantido pelo Driver ADO.NET reaproveitando handshakes TCP caros ao BD. Diferente de Thread Pool que gerencia CPU do servidor.

3. Arquitetura, Segurança e Resiliência

Q6, 10, 24: Resiliência Externa e Filas (Idempotência)

Retry & Circuit Breaker: Use Polly. Falhou 3x? Abre o circuito e para de enviar requests.

Filas e Idempotência: Worker consumiu do SQS mas caiu antes de dar ACK? A mensagem volta. Para não processar a compra duas vezes, verifique a `IdempotencyKey` no Banco. Erros máximos vão para a DLQ (Dead Letter Queue).

Q21, 22: Cancelamento e Saga Pattern

Cancelamento de 1M (CancellationToken): Propague o token da Controller pro HttpClient/EFCore. Usuário abortou o request? Dispara Exception e o banco dá Rollback instantâneo.

Reverter chamadas externas: Sistemas externos não participam do seu Rollback ACID. Use a Saga Pattern (Transações de Compensação): Se o passo C falhar, chame a API de estorno do Passo B.

Q7, 8, 18, 19: Proteger API e Observabilidade

AuthN vs AuthZ: Autenticação (Saber quem é, JWT). Autorização (Saber o que pode fazer, Roles/Claims).

Proteção (Rate Limit): WAF na frente. Configurar Rate Limiting por IP ou `UserId` na API para barrar DDOS. Métricas baseadas em OpenTelemetry e Logs com Serilog apontam quem é o agressor.

Q11, 12, 14: Cloud e Padrões (Patterns)

Terraform: Infraestrutura como Código. Declara via arquivo e StateFile.

Efêmeros: AWS Lambda, Containers Fargate. Nascem e morrem, exigem arquitetura Stateless.

Patterns Chaves: Strategy (Eliminar IFs), Decorator (Adicionar cache num repository sem tocá-lo), Outbox Pattern (Gravar BD e publicar Evento de forma atômica).

4. Soft Skills & Clean Code (O Diferencial Sênior)

Q13, 31, 32, 36: Testes e Código "Funcional" x Bom

Testes: Documentação viva. Protegem regressão.

Coesão e Acoplamento: O desejável é Alta Coesão (Classe focada) e Baixo Acoplamento (Desconectada de implementações via Interfaces).

Código Bom: Código que funciona ("funcional") não é bom. Código bom é legível, expressa o negócio (DDD), testável e tem manutenibilidade fácil.

Q15, 16, 17, 23: Tarefas chatas, Dívida Técnica e Squads

Tarefas chatas: Maturidade profissional. Entregar o valor necessário. Tentar automatizar o "chato" para o futuro.

Dívida Técnica: Nunca esconder. Visibilidade no Board Jira. Alocar tempo constante ou usar a Regra do Escoteiro para ir limpando as dívidas ao redor das entregas de features novas.
